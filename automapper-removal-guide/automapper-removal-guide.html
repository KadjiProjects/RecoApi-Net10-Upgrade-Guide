Replacing AutoMapper with RECO.Mapping

A generic, LLM-executable guide — applies to any .NET project, any target framework.

Independent of the RecoAPI upgrade playbook. This guide is scoped to exactly one thing: removing AutoMapper and replacing it with the RECO.Mapping library. It is written to apply to any .NET project on any target framework, whether or not that project is also being upgraded to .NET 10.

What this guide assumes about you (the executor)

  • Read/write access to a .NET codebase referencing the AutoMapper NuGet package — Profile classes with CreateMap<>(), IMapper injected via DI, services.AddAutoMapper(...), or .ProjectTo<T>() LINQ usage.
  • No prior knowledge of this specific codebase's mapping layout — every phase starts with a discovery step that produces your work list. This guide never assumes a fixed number of mappings, entities, or projects.
  • The ability to run dotnet build / dotnet test and read their output.

The library you are migrating to

RECO.Mapping — a small, dependency-free library of mapping contracts and a test-time coverage verifier. Its complete, buildable source lives in this repository at ../src/RECO.Mapping/ (7 files, zero package dependencies), with a working demo/test project at ../tests/RECO.Mapping.Tests/ you can build and run right now: 

dotnet build ../RECO.Mapping.sln
dotnet test  ../RECO.Mapping.sln

Two usage tiers, chosen per mapping pair depending on what you own and what C# version you target (full rules in §2):

TierWhen to use it
InterfaceIMappedFrom/IMapsTo/IScalarCopyable/IDualMapped — requires C# 11 / .NET 7+ (static abstract interface members) and that you can add an interface to the type. Gives generic-constraint call sites (TDest.MapFrom(source), source.MapTo()) with zero runtime indirection.
FallbackPlain static/extension methods — works on any C# version, any framework, any type you don't own. MappingVerifier works identically either way; it only needs a Func<TSource,TDest>.

How to execute (instructions for the LLM)

  1. Read §0 Ground rules completely before touching anything.
  2. Execute the sections in numeric order. Each ends with a GATE — do not proceed past a failing GATE; if §7 doesn't cover the failure, stop and report.
  3. §1 Discovery produces a table of every mapping pair — every later section operates on that table.
  4. §3 Migration patterns is the core: a numbered catalog of every AutoMapper feature with the exact replacement code.
  5. Finish with §8 Final checklist as your acceptance gate.

Scope boundary — what this guide does NOT do

  • Does not upgrade your target framework — that's a separate step.
  • Does not redesign your mapped types — behavioral parity, not a data-model refactor.
  • Does not cover mapping libraries other than AutoMapper (though the target pattern suits any of them).

0. Ground rules — read completely before doing anything

0.1 What "replace AutoMapper" means here, precisely

By the end of this guide, in the target codebase:

  • The AutoMapper NuGet package (and companions) is removed from every project.
  • Every Profile-derived class and every CreateMap<>() configuration is gone.
  • Every _mapper.Map<T>(x), _mapper.Map(x, existing), or .ProjectTo<T>() call now calls explicit, compiled C# — a static method implementing a RECO.Mapping interface, a plain static/extension method, or (for ProjectTo specifically) a compiled Expression<Func<TSource,TDest>>.
  • Every mapping pair has at least one test built on MappingVerifier, the direct replacement for AssertConfigurationIsValid().
  • The codebase's behavior is unchanged — a mechanical translation, not a redesign.

0.2 Your codebase is unknown — discover, never assume

This guide is generic on purpose: no baseline entity count, no baseline project count. Every section after §1 operates on the table that section produces for your codebase. Do not skip discovery because "it's probably simple" — AutoMapper's fluent API makes it trivial to hide a custom value resolver, a conditional map, or a ProjectTo call inside a large Profile file.

0.3 Hard rules (do NOT deviate)

  1. DO NOT introduce a different mapping library (Mapperly, TinyMapper, hand-rolled reflection). The target is RECO.Mapping (or plain static methods) — full stop.
  2. DO NOT let a previously-mapped member go unpopulated without a conscious, recorded unmapped-by-design decision.
  3. DO NOT assume AutoMapper's convention-based flattening (e.g. Source.Address.City → Dest.AddressCity) happened the way you'd guess — verify it (§3.14).
  4. DO NOT reuse a compiled mapping method inside an EF Core IQueryable LINQ query that used .ProjectTo<T>() — see §3.15 / §5.2.
  5. DO NOT silently approximate a mapping you can't mechanically translate (DI-dependent resolvers, PreserveReferences(), polymorphic Include<>()) — flag per §5.
  6. DO run every GATE. If a GATE fails and §7 doesn't cover it: stop and report.

0.4 Prerequisites — pick your library tier before starting

The four interface files use static abstract interface members, a C# 11 feature requiring the .NET 7 SDK or later.

Your situationTier
.NET 7, 8, 9, 10, or any framework buildable with a .NET 7+ SDK and <LangVersion>11</LangVersion> or laterInterface tier — use all four interfaces as shipped.
.NET 6 or earlier, .NET Standard, or .NET Framework (4.x)Fallback tier — skip the four interfaces; use plain static/extension methods. MappingVerifier is unaffected either way.

Mixed solutions may use different tiers per project — that's fine, tiers interoperate at the call-site level.

0.5 What "done" looks like

§8 is the authoritative acceptance list. In summary: zero AutoMapper references remain; every mapping pair has explicit code and a passing verifier test; the solution builds with no new warnings; pre-existing functional tests still pass unmodified.

2.2 Interface tier (C# 11 / .NET 7+) — use everything as-is

2.3 Fallback tier (pre-C# 11 / .NET Framework / .NET Standard) — swap 5 files

  1. Delete (or don't add): IMappedFrom.cs, IMapsTo.cs, IScalarCopyable.cs, IDualMapped.cs — will not compile below C# 11.
  2. Replace MappingExtensions.cs with this delegate-based version (compiled and run standalone to confirm it works, before publication):
namespace RECO.Mapping;

/// <summary>
/// Collection helpers over plain mapping delegates. Fallback-tier equivalent of the
/// interface-based MapFromAll/MapToAll — takes the mapping function explicitly
/// instead of resolving it through a static interface member, so it works on any
/// C# version and with types you don't own.
/// </summary>
public static class MappingExtensions
{
    /// <summary>Maps every element of <paramref name="sources"/> with <paramref name="map"/>.</summary>
    public static List<TDest> MapAll<TSource, TDest>(
        this IEnumerable<TSource> sources, Func<TSource, TDest> map)
        => sources.Select(map).ToList();
}

Call sites become sources.MapAll(PersonMapper.ToDto) instead of sources.MapFromAll<TDest, TSource>().

  1. Keep unchanged: RECO.Mapping.csproj (retarget only), MappingVerificationException.cs, MappingVerifier.cs — neither verification file uses static abstract members; they use only reflection and a plain Func<,> delegate, available since .NET Framework 4.5.

3. Migration patterns — every AutoMapper feature and its exact replacement

Work through your discovery table row by row. Nested/collection patterns (§3.12–3.13) are usually applied around whichever base pattern (§3.1–3.11) the pair's top-level shape uses. Each pattern gives both library tiers' code where the tier changes the shape.

3.1 Simple map — CreateMap<Source, Dest>(), no modifiers

Write every assignment explicitly — do not rely on name-matching happening automatically.

Interface tier (you own Dest's source):

public sealed class PersonDto
{
    public int Id { get; set; }
    public string FirstName { get; set; } = "";
    public string LastName { get; set; } = "";
}

public sealed class PersonEntity : IMappedFrom<PersonEntity, PersonDto>
{
    public int Id { get; set; }
    public string FirstName { get; set; } = "";
    public string LastName { get; set; } = "";

    public static PersonEntity MapFrom(PersonDto source) => new()
    {
        Id = source.Id,
        FirstName = source.FirstName,
        LastName = source.LastName,
    };
}

// call site: var entity = PersonEntity.MapFrom(dto);

Fallback tier (or you don't own either type):

public static class PersonMapper
{
    public static PersonEntity ToEntity(this PersonDto source) => new()
    {
        Id = source.Id,
        FirstName = source.FirstName,
        LastName = source.LastName,
    };
}

// call site: var entity = dto.ToEntity();

Both are equally valid; the interface tier only buys generic-constraint call sites in pipeline code (§4.3).

3.2 Ignored member — .ForMember(d => d.X, opt => opt.Ignore())

Don't assign X. Record nameof(Dest.X) — you'll pass it as unmappedByDesign to MappingVerifier in §6, proving it was intentionally skipped, not forgotten.

3.3 Custom member expression — .ForMember(d => d.X, opt => opt.MapFrom(s => expr))

FullName = $"{source.FirstName} {source.LastName}",

If expr is non-trivial, extract a small private static helper in the same mapping class.

3.4 Value resolver class — .ForMember(d => d.X, opt => opt.MapFrom<SomeResolver>())

Find the resolver's Resolve(...) body. No constructor dependencies → inline like §3.3. Has injected dependencies → see §5.1 (DI-dependent mapping services).

3.5 ReverseMap()

Interface tier:

public sealed class PersonEntity : IDualMapped<PersonEntity, PersonDto>
{
    // ... properties ...
    public static PersonEntity MapFrom(PersonDto source) => new() { /* ... */ };
    public PersonDto MapTo() => new() { /* ... */ };
}

IDualMapped<TSelf,TOther> is exactly IMappedFrom<TSelf,TOther> + IMapsTo<TOther>.

Fallback tier:

public static class PersonMapper
{
    public static PersonEntity ToEntity(this PersonDto source) => new() { /* ... */ };
    public static PersonDto ToDto(this PersonEntity entity) => new() { /* ... */ };
}

If ReverseMap() had its own additional modifiers for the reverse direction, apply them only to that direction's method.

3.6 ConvertUsing(...) — whole-type custom conversion

The simplest case in disguise — write MapFrom/MapTo/ToEntity/ToDto with a single expression matching the conversion function. Nothing else to translate.

3.7 ConstructUsing(...) — custom construction, then normal mapping continues

public static PersonEntity MapFrom(PersonDto source) => new PersonEntity(source.Id) // custom ctor call
{
    FirstName = source.FirstName, // normal member mapping continues
    LastName = source.LastName,
};

3.8 Condition(...) — member mapped only if a predicate holds

Constructing a new instance:

Discount = source.IsPremiumMember ? source.Discount : 0m,

Updating an existing instance where the false case must preserve the existing value — this is actually the in-place-update pattern (§3.10):

public void CopyScalarsTo(PersonEntity target)
{
    if (IsPremiumMember) target.Discount = Discount; // else target.Discount is untouched
    target.FirstName = FirstName; // unconditional members still copy normally
}

3.9 NullSubstitute(...)

DisplayName = source.PreferredName ?? "Anonymous",

3.10 In-place update — _mapper.Map(source, existingDestination) (two-argument overload)

This is IScalarCopyable<T>.CopyScalarsTo — updating a tracked instance without disturbing navigation/relationship properties.

Interface tier:

public sealed class PersonEntity : IScalarCopyable<PersonEntity>
{
    public int Id { get; set; }
    public string FirstName { get; set; } = "";
    public List<OrderEntity> Orders { get; set; } = []; // navigation — must NOT be touched

    public PersonEntity CloneScalars() => new() { Id = Id, FirstName = FirstName };

    public void CopyScalarsTo(PersonEntity target)
    {
        target.Id = Id;
        target.FirstName = FirstName;
        // target.Orders is intentionally left untouched.
    }
}

// call site (was: _mapper.Map(incoming, tracked)):
incoming.CopyScalarsTo(tracked);

Fallback tier:

public static class PersonMapper
{
    public static PersonEntity CloneScalars(this PersonEntity source) => new()
    {
        Id = source.Id, FirstName = source.FirstName,
    };

    public static void CopyScalarsTo(this PersonEntity source, PersonEntity target)
    {
        target.Id = source.Id;
        target.FirstName = source.FirstName;
        // target.Orders is intentionally left untouched.
    }
}

If the old code also snapshotted "before" state prior to overwriting, that snapshot call is CloneScalars().

3.11 Self-mapping / deep clone — CreateMap<T, T>()

If used to snapshot scalar state, it's CloneScalars() (§3.10). For a genuine deep clone:

public PersonEntity DeepClone() => new()
{
    Id = Id, FirstName = FirstName,
    Orders = [.. Orders.Select(o => o.DeepClone())],
};

For record types, prefer a with expression for the shallow case — records give you this for free.

3.12 Nested / complex-type member

Give the nested pair its own mapping first (leaf types before parents), then the parent calls it directly:

// AddressDto/AddressEntity already have their own MapFrom/MapTo from §3.1 or §3.5.
public static PersonEntity MapFrom(PersonDto source) => new()
{
    Id = source.Id,
    Address = source.Address is null ? null : AddressEntity.MapFrom(source.Address),
    // fallback tier: Address = source.Address?.ToEntity(),
};

3.13 Collection of a nested mapped pair

// Interface tier:
Lines = source.Lines.MapFromAll<LineEntity, LineDto>(),

// Fallback tier:
Lines = source.Lines.MapAll(LineMapper.ToEntity),

3.14 Automatic member-name "flattening"

The single most common source of a silent regression in this kind of migration. AutoMapper maps Source.Address.City to Dest.AddressCity with zero explicit configuration — nothing in the Profile file tells you. It compiles fine either way; a forgotten flattened property silently stays default at runtime.

For every destination property with no same-named source property, check whether it matches a concatenation of the source's nested paths:

AddressCity = source.Address?.City ?? "",
AddressPostalCode = source.Address?.PostalCode ?? "",

Check every destination property in every pair — it's exactly the "simple" mappings (§3.1, no explicit ForMember) that are most likely relying entirely on this invisible convention.

3.15 ProjectTo<TDest>() on an IQueryable<TSource> — LINQ-to-SQL projection

Fundamentally different from every other pattern here. ProjectTo builds an expression tree translated to SQL before materialization. A compiled method call is opaque to that translator — it throws, or silently forces full materialization first (a serious performance regression).
// was: query.ProjectTo<PersonDto>(mapperConfig)
var results = query.Select(p => new PersonDto
{
    Id = p.Id,
    FullName = p.FirstName + " " + p.LastName, // SQL-translatable string concat
}).ToList();

If the shape repeats across queries, extract a static expression tree:

public static class PersonProjections
{
    public static Expression<Func<Person, PersonDto>> ToDto => p => new PersonDto
    {
        Id = p.Id,
        FullName = p.FirstName + " " + p.LastName,
    };
}

// call site: query.Select(PersonProjections.ToDto)

Only use SQL-translatable expressions inside the projection body. If code execution is genuinely needed, materialize first and map in memory — a conscious tradeoff, not an accident.

4.3 Generic pipeline / service classes parameterized by <TSource, TDest>

If discovery flagged a class taking IMapper internally keyed by type parameters:

You own both type families everywhere this class is used — constrain the class itself:

public class ActionProcessor<TDataModel, TDbModel>
    where TDbModel : IDualMapped<TDbModel, TDataModel>
{
    // no IMapper constructor parameter anymore
    private void Save(TDataModel data)
    {
        TDbModel dbModel = TDbModel.MapFrom(data);
    }

    private TDataModel Load(TDbModel dbModel) => dbModel.MapTo();
}

You don't own the types everywhere it's used — inject plain delegates instead:

public class ActionProcessor<TDataModel, TDbModel>
{
    private readonly Func<TDataModel, TDbModel> _toDbModel;
    private readonly Func<TDbModel, TDataModel> _toDataModel;

    public ActionProcessor(Func<TDataModel, TDbModel> toDbModel, Func<TDbModel, TDataModel> toDataModel)
    {
        _toDbModel = toDbModel;
        _toDataModel = toDataModel;
    }
}

// composition root, one line per concrete type pair:
services.AddScoped<ActionProcessor<PersonDto, PersonEntity>>(sp =>
    new ActionProcessor<PersonDto, PersonEntity>(PersonMapper.ToEntity, PersonMapper.ToDto));

Either way, every previously-dynamic mapping call becomes visible and compile-time checked at its instantiation site.

5. Special cases — slow down and think, don't mechanically apply

These require a design decision. Flag them rather than guessing — a wrong guess here is silent and expensive precisely because it usually still compiles and often still runs.

5.1 DI-dependent mappings (custom value resolver with injected dependencies)

public sealed class PersonMappingService(ICodeLookupService lookup, TimeProvider clock)
{
    public PersonDto ToDto(PersonEntity entity) => new()
    {
        Id = entity.Id,
        CountryName = lookup.NameFor(entity.CountryCode),
        SnapshotTakenUtc = clock.GetUtcNow(),
    };
}

Verified the same way as everything else — construct with test doubles, pass the method as the delegate:

var service = new PersonMappingService(fakeLookup, fakeClock);
MappingVerifier.AssertAllMembersMapped<PersonEntity, PersonDto>(service.ToDto);

5.2 Inheritance / polymorphic mapping — .Include<Derived, DerivedDto>()

No automatic equivalent — write one mapping method per concrete pair, dispatch with a type switch:

BaseDto dto = source switch
{
    Derived1 d1 => Derived1Dto.MapFrom(d1),
    Derived2 d2 => Derived2Dto.MapFrom(d2),
    _ => BaseDto.MapFrom(source),
};

Consider a reflection-based test that every subtype of Base has a case in the switch, so a forgotten case fails a test rather than silently falling through at runtime.

5.3 Circular / self-referencing object graphs — PreserveReferences()

Hand-written recursive mapping has no cycle protection — a true cycle will stack-overflow. Most common resolution: map the back-reference as an ID only (ManagerId = source.Manager?.Id). If the full nested object is genuinely required both ways: cap recursion depth or maintain your own visited-identity map through the recursive calls.

5.4 Global naming conventions beyond flattening

If SourceMemberNamingConvention/DestinationMemberNamingConvention was non-default (e.g. snake_case → PascalCase), this is a more aggressive version of §3.14 — verify every destination property against the source's actual member names, not just its structure.

6. Verification — one test per mapping pair, per direction

MappingVerifier.AssertAllMembersMapped<TSource,TDest>(map, params unmappedByDesign) is the direct replacement for AssertConfigurationIsValid(). It takes any Func<TSource,TDest> — no interface requirement, works identically for both tiers and DI-dependent services.

6.1 The basic test shape

[Test]
public void PersonDto_maps_to_PersonEntity_completely()
{
    MappingVerifier.AssertAllMembersMapped<PersonDto, PersonEntity>(
        PersonEntity.MapFrom,
        /* unmappedByDesign: */ nameof(PersonEntity.Tags));
}

[Test]
public void PersonEntity_maps_to_PersonDto_completely()
{
    MappingVerifier.AssertAllMembersMapped<PersonEntity, PersonDto>(e => e.MapTo());
}

The unmappedByDesign list must exactly match the members recorded as ignored in §3.2 for that pair and direction.

6.2 In-place update pairs (§3.10)

[Test]
public void CopyScalarsTo_covers_every_scalar_member()
{
    MappingVerifier.AssertAllMembersMapped<PersonEntity, PersonEntity>(
        source => { var target = new PersonEntity(); source.CopyScalarsTo(target); return target; },
        nameof(PersonEntity.Orders));
}

6.3 DI-dependent mapping services (§5.1)

var service = new PersonMappingService(new FakeCodeLookupService(), FakeTimeProvider());
MappingVerifier.AssertAllMembersMapped<PersonEntity, PersonDto>(service.ToDto);

6.4 ProjectTo-replaced LINQ expressions (§3.15)

[Test]
public void PersonProjection_covers_every_destination_member()
    => MappingVerifier.AssertAllMembersMapped<Person, PersonDto>(PersonProjections.ToDto.Compile());

6.5 Inheritance dispatch (§5.2)

One coverage test per concrete derived pair, plus — if built — the reflection-based "every subtype has a case" test.

8. Final checklist and change summary

The single acceptance pass — summarizes every GATE from §0 through §6. A clean build proves the code compiles, not that every mapping was migrated.

Acceptance checklist

  • Discovery (§1) is complete: every CreateMap, call site, global option recorded.
  • Library is in place (§2): correct tier chosen per project; builds with 0 errors.
  • Every mapping pair has explicit replacement code (§3): every member assigned or recorded unmapped-by-design; every §3.14 candidate checked.
  • Every call site rewritten (§4): the AutoMapper/IMapper grep returns nothing.
  • Every special case resolved by conscious decision (§5): DI-dependent mappings are services; polymorphic mappings dispatch explicitly; cycles broken deliberately.
  • Every mapping pair has a passing MappingVerifier test (§6), ignore lists matching §3.2 exactly.
  • Solution builds with no new warnings.
  • Pre-existing functional/integration tests still pass, unmodified.
  • AutoMapper package reference is gone from every project.
